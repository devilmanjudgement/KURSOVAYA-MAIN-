Для хранения данных веб‑приложения было спроектировано и реализовано реляционное хранилище kgusport.db, работающее на базе системы SQLite. Это решение выбрано благодаря простоте развёртывания, отсутствию необходимости в установке отдельного серверного ПО и удобной интеграции с сервером, написанным на Node.js и Express.js.

База данных строится по реляционной модели и включает девять таблиц, связанных между собой отношениями «один‑ко‑многим», что позволяет поддерживать целостность данных и избегать их дублирования.

Таблица users предназначена для хранения информации обо всех участниках системы — студентах, преподавателях и администраторах. В ней содержатся поля: id (первичный ключ), name, login, password, role, group_name, avatar, health_doc и student_id. По значению поля role определяется набор доступных пользователю функций.

Таблица sections содержит сведения о спортивных секциях: id, title, coach_id, place, color, image, description, max_students. Поле coach_id является внешним ключом, связывающим секцию с преподавателем из таблицы users. Через эту таблицу формируется каталог секций на клиентской части приложения.

Таблица bookings фиксирует заявки студентов на участие в секциях. Хранятся: идентификатор записи bookingId, идентификатор секции sectionId, имя студента user, дата подачи date, прикреплённый документ docType и текущий статус заявки status (pending / approved / cancelled).

Таблица schedule предназначена для хранения расписания занятий. Содержит поля day_of_week, time, а также внешние ключи section_id и coach_id, что позволяет формировать индивидуальное расписание как для студентов, так и для преподавателей.

Таблица messages реализует встроенный мессенджер. Каждое сообщение хранит идентификатор отправителя sender_id и получателя receiver_id (оба являются внешними ключами на таблицу users), текст сообщения, признак прочтения read и метку времени created_at.

Таблица attendance служит журналом посещаемости занятий. Хранит имя студента, идентификатор и название секции, идентификатор тренера, дату занятия и признак присутствия (present: 1 — присутствовал, 0 — отсутствовал).

Таблица student_registry содержит официальный реестр студентов, загружаемый администратором в формате CSV. Первичным ключом служит student_id — номер студенческого билета. Таблица используется для верификации данных при регистрации.

Таблица pending_registrations хранит заявки на создание учётной записи, поданные через форму регистрации, до их рассмотрения администратором. Содержит данные формы (ФИО, логин, email, student_id), IP‑адрес подачи, статус заявки и причину отклонения при необходимости.

Таблица section_posts хранит объявления, публикуемые тренером внутри секции. Поле section_id является внешним ключом на таблицу sections.

Между таблицами установлены следующие ключевые связи: один преподаватель (users) может вести несколько секций (sections) и иметь несколько записей расписания (schedule); каждая секция может содержать множество заявок (bookings), записей расписания (schedule) и объявлений (section_posts); в мессенджере каждое сообщение (messages) связано с отправителем и получателем из таблицы users.

Создание всех таблиц выполняется автоматически при запуске приложения командами CREATE TABLE IF NOT EXISTS, что исключает необходимость ручного администрирования базы данных.
