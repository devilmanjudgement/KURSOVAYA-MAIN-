# КГУ СПОРТ — материалы для дипломной работы

## 1. Назначение и предметная область

**КГУ СПОРТ** — веб-платформа записи студентов Кыргызского Государственного Университета на спортивные секции.

Решает три прикладные задачи:
- Студент находит секцию, видит расписание/тренера/места и подаёт заявку.
- Тренер ведёт секцию: список заявок, посещаемость, расписание, объявления для своих, мессенджер.
- Администратор управляет всей системой: пользователи, реестр студентов, модерация заявок на регистрацию, журнал событий безопасности.

Целевая аудитория: вуз с фиксированным контингентом студентов (есть официальный реестр), несколько десятков секций, до нескольких тысяч пользователей.

## 2. Архитектура и технологический стек

Архитектура — классическая клиент-серверная двухзвенка с REST API:

| Слой | Технология | Порт | Роль |
|---|---|---|---|
| Frontend (SPA) | React 19 + Vite 7 | 5000 | UI, маршрутизация, локализация |
| Backend (REST) | Node.js + Express 5 | 3001 | Бизнес-логика, безопасность, файлы |
| База данных | SQLite (better-sqlite3) | — | Хранение всех сущностей |
| Доп. компоненты | Multer (загрузка файлов), SSE (push-уведомления) | — | Файлы и реалтайм |

**Почему такой стек подходит для диплома:**
- React + Vite даёт быструю отрисовку и горячую перезагрузку при разработке (HMR).
- Express 5 — минималистичный фреймворк, легко показать обработчики и middleware.
- SQLite через `better-sqlite3` — синхронный нативный драйвер, очень быстрый на операциях «много мелких чтений», не требует отдельного сервера БД.
- Vite в dev-режиме проксирует `/api` и `/images` на бэкенд (`vite.config.js`) — фронтенд ходит на относительные URL, что устраняет проблему CORS и упрощает деплой.

**Структура каталогов:**
```
web-prilojenie/
  src/                 # React-приложение (~5,3 тыс. строк)
    App.jsx, Login.jsx, Register.jsx, Home.jsx, Section.jsx,
    BookingConfirm.jsx, Profile.jsx, TeacherPanel.jsx,
    Schedule.jsx, ScheduleEdit.jsx, SearchPage.jsx,
    SectionsList.jsx, Chat.jsx, Navbar.jsx, AdminPanel.jsx,
    Attendance.jsx, TeacherBookings.jsx, Success.jsx,
    contexts/LangContext.jsx
  server/
    index.js           # Единый сервер: API + безопасность + сид (~970 строк)
    images/            # Загруженные аватары, обложки секций, справки
    kgusport.db        # SQLite-файл
  vite.config.js       # Прокси /api и /images → :3001
```

## 3. Модель данных

База содержит 9 таблиц, связи на FOREIGN KEY:

1. **users** — `id, name, login, password, role (admin/coach/student), group_name, avatar, health_doc, student_id`
2. **sections** — `id, title, coach_id → users, place, color, image, description, max_students`
3. **bookings** — `bookingId, sectionId, user, date, docType, status (pending/approved/cancelled)`
4. **schedule** — `id, day_of_week, time, section_id → sections, coach_id`
5. **messages** — `id, sender_id, receiver_id, text, read, created_at`
6. **attendance** — `id, student_name, section_id, section_title, coach_id, date, present`
7. **student_registry** — официальный реестр (CSV-импорт): `student_id, first_name, last_name, middle_name, group_name`
8. **pending_registrations** — очередь заявок: `id, login, password, ФИО, email, ip, status, rejection_reason`
9. **section_posts** — объявления тренера «для своих»: `id, section_id, coach_id, text, created_at`

**Особенности модели:**
- В `bookings.user` хранится имя (TEXT), а не FK на `users.id` — это упрощает отображение, но требует учитывать в JOIN-ах (используется `JOIN users u ON u.name = b.user`).
- `sections.id` — строковый ключ вида `sec_<timestamp>` (генерируется на сервере), что облегчает дебаг и читабельность.
- Миграции выполняются «на лету» через `CREATE TABLE IF NOT EXISTS` и `ALTER TABLE … ADD COLUMN` в try/catch — БД сама обновляется при старте.

## 4. Основной функционал

### 4.1. Авторизация и регистрация
- **Вход** по логину/паролю, без сессии — фронтенд хранит профиль в `localStorage`.
- **Регистрация студентов через очередь модерации** (двухступенчатая):
  1. Студент вводит ФИО + логин/пароль/email.
  2. Email-верификация 4-значным кодом (TTL 10 минут, в текущей версии — заглушка, код выводится в консоль и в поле `_devCode`; легко заменяется на реальный SMTP).
  3. Создаётся запись в `pending_registrations`, аккаунт **не создаётся сразу**.
  4. Администратор сверяет ФИО с официальным CSV-реестром и одобряет/отклоняет.
  5. При одобрении автоматически создаётся `users`-запись.
- **Тренеры регистрируются только через сид/админа** — публичной формы нет (защита от самопроизвольного появления преподавателей).
- При первом запуске сид создаёт одного админа (`admin/admin123`), несколько тренеров и тестовых студентов.

### 4.2. Секции и записи
- Карточки секций на главной: фон, обложка, тренер, место, заполненность.
- Просмотр секции: описание, расписание, кнопка «Записаться».
- Создание/редактирование секции тренером и админом, загрузка обложки (Multer, до 5 МБ).
- Запись (`POST /api/bookings`) с тремя проверками:
  1. Секция существует.
  2. Студент не записан повторно (status ≠ cancelled).
  3. Не превышен лимит мест (`max_students`).
- Тренер видит заявки своей секции и подтверждает/отклоняет.

### 4.3. Расписание
- Просмотр: красивая раскладка по дням недели.
- Редактирование: тренер добавляет/удаляет слоты для своих секций.

### 4.4. Посещаемость
- Тренер отмечает явку студентов по конкретной дате.
- Студент видит свою историю посещений в профиле.

### 4.5. Мессенджер
- 1-на-1 переписка тренер ↔ студент.
- Эмодзи, индикатор «прочитано», красный бейдж непрочитанных в навбаре.
- **Server-Sent Events (SSE)** для уведомлений в реальном времени о смене статуса заявки и новых сообщениях — без WebSocket-инфраструктуры.

### 4.6. Объявления секции «Для своих»
- Тренер публикует посты, видимые только записанным в секцию.
- Защита: коллбэк проверяет, что `coach_id` действительно владелец секции.

### 4.7. Профиль
- Редактирование имени/группы/аватара.
- Смена пароля с проверкой старого.
- Загрузка справки о здоровье (PNG/PDF, до 5 МБ).

### 4.8. Панель администратора
- Полноэкранная страница со вкладками:
  - **Пользователи** — список, удаление с каскадом.
  - **Секции** — все секции с тренерами.
  - **Бронирования** — все заявки.
  - **Заявки на регистрацию** — модерация.
  - **Реестр** — загрузка CSV (UTF-8 BOM, разделитель `;` или `,`, авто-определение).
  - **Журнал безопасности** — последние 200 событий (логины, ошибки, превышения лимитов).

### 4.9. Локализация
- Два языка (русский по умолчанию, английский), переключение в навбаре.
- Все строки централизованы в `src/contexts/LangContext.jsx` (объект `T`).
- Хук `useLang()` отдаёт `t(key)` — функцию-переводчик.
- Выбор сохраняется в `localStorage`.

## 5. Что и как оптимизировано

### 5.1. База данных и серверная часть
- **`better-sqlite3` вместо асинхронного драйвера.** Синхронный нативный драйвер обходит overhead очереди событий Node.js на быстрых in-process чтениях; на типовых выборках это в 2–5 раз быстрее `sqlite3`.
- **Подготовленные запросы (`db.prepare(...).run/get/all`).** План запроса кэшируется один раз, далее выполняется без повторного парсинга — ускорение и одновременно защита от SQL-инъекций.
- **Транзакции для каскадного удаления.** Удаление секции (`DELETE /api/sections/:id`) выполняется в `db.transaction(() => { … })()`, что гарантирует атомарность и заметно быстрее серии независимых `DELETE`.
- **Агрегаты на уровне SQL.** Например, в `/api/sections` количество записанных студентов считается коррелированным подзапросом `(SELECT COUNT(*) … WHERE status='approved')`, а не отдельным круговоротом запросов из кода.
- **`GROUP BY` для устранения дублей.** В `/api/sections/:id/enrolled` используется `GROUP BY u.id` — дубли студентов не приходят на фронт.
- **In-memory rate-limit и журнал событий** — без отдельного Redis: достаточно для уровня вуза, нет внешней зависимости.
- **Очистка устаревших записей по таймеру.** `setInterval` каждые 60 с чистит карту rate-limit (`>120 с` старше) и каждые 60 мин — карту IP-лимита регистраций (`>24 ч`). Карты не растут бесконечно.
- **SSE вместо WebSocket** — меньше накладных расходов и не требует отдельного шлюза/прокси с поддержкой `Upgrade`.

### 5.2. Клиентская часть
- **Vite c esbuild** — холодный старт ~250 мс, мгновенный HMR.
- **Проксирование `/api` и `/images`** в `vite.config.js` устраняет CORS на dev и позволяет фронтенду использовать относительные URL — те же ссылки работают и в production.
- **Статика картинок** отдаётся `express.static` с заголовками кэша по умолчанию — браузер не перекачивает аватары при каждой навигации.
- **Минимум сторонних UI-библиотек** — иконки `lucide-react` + собственный CSS, отсутствие тяжёлых наборов компонентов (MUI/Antd) даёт маленький бандл.
- **localStorage-стейт** — профиль и язык не запрашиваются с сервера повторно, страницы открываются мгновенно.

### 5.3. Архитектурные оптимизации
- **Единый файл сервера (`index.js`)** — для дипломного проекта это плюс: всю API-поверхность видно в одном месте, легко описывать и защищать. Для production-расширения в дипломе можно описать «направление развития: разбиение на модули по доменам».
- **Файлы и БД лежат локально** — нулевая стоимость инфраструктуры в малой эксплуатации.

## 6. Защита (раздел «Информационная безопасность» в дипломе)

В системе реализовано шесть слоёв защиты, отвечающих основным угрозам OWASP Top 10:

### 6.1. Защита от SQL-инъекций (OWASP A03 — Injection)
- **Все** запросы — параметризованные (`?`-плейсхолдеры в `db.prepare`). Конкатенации SQL-строк с пользовательским вводом нет нигде.
- Дополнительно входные строки проходят `sanitize()` (см. ниже).

### 6.2. Санитизация ввода (XSS, A03)
```js
function sanitize(str) {
  return str.replace(/[<>'"`;\\]/g, "").trim().slice(0, 500);
}
```
- Удаляет потенциально опасные символы (HTML-теги, кавычки, бэктики, обратный слеш, точку с запятой).
- Обрезает длину до 500 символов — защита от «гигантских payload’ов» (DoS на уровне БД).

### 6.3. Защитные HTTP-заголовки (A05 — Security Misconfiguration)
Глобальный middleware:
- `X-Content-Type-Options: nosniff` — блокирует MIME-sniffing.
- `X-Frame-Options: DENY` — защита от clickjacking.
- `X-XSS-Protection: 1; mode=block` — устаревший, но безвредный legacy-флаг.
- `Referrer-Policy: strict-origin-when-cross-origin` — не утекает путь во внешние ссылки.

### 6.4. Rate Limiting (A04 — Insecure Design / brute-force)
- Универсальный middleware `rateLimit(windowMs, max)` на основе IP+URL.
- Конкретные лимиты:
  - `/api/login` — 10 попыток/мин (защита от подбора пароля).
  - `/api/register` — 10/мин.
  - `/api/auth/send-code` — 5/мин (защита от спама email).
  - `/api/auth/verify-code` — 10/мин.
- **Отдельный IP-лимит на регистрации**: не более **3 заявок с одного IP за 24 часа**, чтобы один человек не забил очередь модерации.
- При превышении — 429 Too Many Requests + запись в журнал безопасности.

### 6.5. Контроль загружаемых файлов (A04, A08 — Software/Data Integrity)
- Multer с жёсткими ограничениями:
  - Размер ≤ 5 МБ.
  - MIME-фильтр: только `image/jpeg|png|gif|webp` и `application/pdf` (а для медсправок — только PNG и PDF).
- Имя файла генерируется сервером (`fieldname_<timestamp>_<rand><ext>`) — пользователь не может перезаписать чужой файл.
- CSV-реестр читается из памяти (`memoryStorage`), на диск не сохраняется.

### 6.6. Журнал событий безопасности (A09 — Logging & Monitoring)
- Кольцевой буфер на 200 записей, доступ через `/api/admin/logs`.
- Логируются: успешные логины (INFO), неуспешные логины (WARN), превышения rate-limit, ошибки верификации кодов, заявки на регистрацию.
- Уровни `WARN/ERROR` дополнительно дублируются в `console.log`.

### 6.7. Прочие меры
- Хеши паролей и поле `password` **никогда не возвращаются** клиенту: на ответе всегда делается `const { password: _pw, ...safe } = user`.
- Валидация ролей при действиях: например, добавлять/удалять объявления секции может только тренер-владелец секции (`section.coach_id === coach_id`).
- Каскадное удаление в транзакции исключает «висящие» строки и нарушение FK.
- Лимит JSON-тела: `express.json({ limit: "2mb" })`.

### 6.8. Что честно сказать в разделе «Ограничения и направления улучшения»
Для академической честности (и чтобы вопросов на защите было меньше) стоит явно перечислить, что **сейчас не реализовано** и почему это допустимо в рамках дипломного прототипа:
- Пароли хранятся в **открытом виде** — для production требуется bcrypt/argon2.
- Нет JWT/сессий: авторизация на роуты не проверяется на сервере — только разделение интерфейсов на фронте. Решение: добавить middleware `requireAuth(role)` на основе подписанного JWT.
- Email-код — заглушка (отдаётся в ответе как `_devCode`); требуется подключение SMTP/SendGrid/Mailgun.
- HTTPS не настроен в самом приложении — выносится на уровень reverse-proxy/деплоя (см. ниже).
- CSRF-защита не нужна, пока нет cookie-сессий, но появится одновременно с ними.

В дипломе это лучше формулировать как «направление дальнейшего развития».

## 7. Развёртывание для общего пользования

Здесь у тебя есть три уровня сложности — выбирай по тому, как и где будет «жить» проект.

### 7.1. Самый быстрый путь — публикация в Replit (1 клик)
В файле `.replit` уже описана конфигурация деплоя (VM-target):
```
[deployment]
deploymentTarget = "vm"
run = ["bash","-c","node web-prilojenie/server/index.js & cd web-prilojenie && npm run build && npx serve dist -l 5000"]
```
Что произойдёт при публикации:
1. Replit соберёт фронтенд (`vite build`) — получится статика в `web-prilojenie/dist`.
2. Запустит бэкенд на :3001 и статический сервер на :5000.
3. Выдаст постоянный домен вида `*.replit.app` с автоматическим HTTPS и health-check.
4. БД и папка `images/` сохраняются в файловой системе VM (важно: VM, не Autoscale — иначе теряется состояние SQLite и файлов).

Подходит для: демонстрации на защите, реальной эксплуатации в одном вузе с десятками-сотнями пользователей.

### 7.2. Деплой на собственный сервер (VPS, например, для университета)

**Подготовка сервера** (Ubuntu 22.04+):
```bash
# Node.js LTS
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs build-essential

# Менеджер процессов
sudo npm install -g pm2

# Reverse proxy и TLS
sudo apt-get install -y nginx certbot python3-certbot-nginx
```

**Развёртывание приложения:**
```bash
git clone <repo> /opt/kgusport
cd /opt/kgusport
npm install
cd web-prilojenie && npm install && npm run build && cd ..

# Запуск под pm2
pm2 start web-prilojenie/server/index.js --name kgusport-api
pm2 startup && pm2 save
```

**Nginx** (`/etc/nginx/sites-available/kgusport`):
```nginx
server {
    server_name sport.example.kg;

    # Статика SPA
    root /opt/kgusport/web-prilojenie/dist;
    index index.html;

    # API → Express
    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;        # важно для SSE
        proxy_read_timeout 1h;
    }

    # Картинки и загруженные файлы
    location /images/ {
        proxy_pass http://127.0.0.1:3001;
    }

    # SPA fallback (React Router)
    location / {
        try_files $uri /index.html;
    }
}
```
TLS:
```bash
sudo certbot --nginx -d sport.example.kg
```

**Что важно для продакшена:**
- Внутри Express прописано `app.set("trust proxy", 1)` стоит добавить, чтобы `req.ip` корректно брался из `X-Forwarded-For`, иначе rate-limit будет считать всех под одним IP nginx.
- Бэкенд должен слушать только `127.0.0.1:3001`, не `0.0.0.0` — наружу торчит только 80/443.
- Резервное копирование БД: `cron`-задача, копирующая `kgusport.db` через `sqlite3 kgusport.db ".backup '/backups/kgusport-$(date +\%F).db'"` (онлайн-бэкап без блокировки).
- Логи pm2 (`pm2 logs`) или редирект в `/var/log/kgusport/`.

### 7.3. Контейнеризация (Docker), если нужна повторяемость

`Dockerfile`:
```dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY web-prilojenie/package*.json web-prilojenie/
RUN cd web-prilojenie && npm ci
COPY . .
RUN cd web-prilojenie && npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app /app
EXPOSE 3001 5000
VOLUME ["/app/data", "/app/web-prilojenie/server/images"]
CMD ["sh","-c","node web-prilojenie/server/index.js & npx serve web-prilojenie/dist -l 5000"]
```
Хранилище БД и картинок выносится в named volume — переустановка контейнера не теряет данные.

### 7.4. Чек-лист «готов к боевой эксплуатации»
1. HTTPS включён (Replit делает сам, на VPS — Let’s Encrypt).
2. Сменены пароли по умолчанию (`admin/admin123` и тестовые тренеры/студенты).
3. Загружен реальный CSV-реестр студентов через админ-панель.
4. Настроен SMTP вместо заглушки email-кодов.
5. Включён ежедневный бэкап `kgusport.db`.
6. Заведён мониторинг (минимум — `pm2 monit` или `uptime-robot` на главную страницу).

## 8. Что можно вынести в раздел «Дальнейшее развитие»

Эти пункты хорошо смотрятся на защите как «продуманное направление масштабирования»:

1. **Хеширование паролей** (bcrypt) и переход на JWT/сессии с защитой роутов на сервере.

5. **Импорт расписания из CSV/1С**, синхронизация со студенческим деканатом.
6. **Аналитика для администрации:** загруженность секций, посещаемость по группам, отчёты в PDF/Excel.
7. **Мобильное приложение** на той же API (PWA или React Native).
8. **CI/CD**: GitHub Actions → автодеплой на Replit/VPS, прогон линтера и тестов на каждом push.
